References

General

FAP_REQ_GEN_01 Creating object from specfication

Framework shall support creation object from specfication

FAP_UC_001 Making changes in the object

There was a spec for object creation provided.
Object was created from the spec.
Some changes get required for the object.
There is mechanism of making change from dedicated change spec

FAP_UC_002 Creating env root from the spec

Preconditions

  • N/A

Steps

  • Clien specifies spec for FAP environment
  • Client creates environment with spec

Result

  • System creates environment automatically accordig with spec

Notes

  • In ver 0.0.1-3 the env is not created automatically but only on "manually" creating root from the correspondig node of spec

FAP_UC_003 Creating environment with dynamic number of objects

Preconditions

  • N/A

Action

  • Client creates environment where the number of objects can be dynamic. The objects withis the environment depends all other objects.
  • Client adds new object and connect all currently existing object wiht new one.

Reaction

  • System allows the client to create new connections without using static connections names.

Notes

  • This case is quite often in environments with many identical objects that connects all together.

FAP_UC_004 USER resets system

User creates system, runs it and then reset. System goes into initial state.

NOTE: Currently this use-case seems invalid because system's chromo can be chanded during the run process (for instance by system's controller). So, no any information of initial state can be presented. Seems we need to reconsider the concept of controlling of system.

States

FAP_REQ_STA_01 Transition by default

If no transion function nor operation is specified for the state then there shall the following default transition be done:

  • Copying the first input current value if it is compatible to state.

NOTE: Earlier one more default transition was used: copying current value of the state to new value. Denied currently.

FAP_REQ_STA_02 Formatting

State shall include APIs for formatting to string and from string

NOTES: To String formatting is required for logging. From string formatting is required for specifying initial values for the state (ref the section called “FAP_REQ_SPC_01 Initial value of state”)

Object specification

General

Refer to the section called “FAP_REQ_GEN_01 Creating object from specfication”

FAP_REQ_SPC_01 Initial value of state

There shall be possibility to specify initial value of state in object specification

FAP_REQ_SPC_02 State incapsulaton

Specification shall describe state with high level of incapsulation.

NOTE: this conforms with approach of representation the system as a set of state machines.

FAP_REQ_SPC_03 State named inputs

Specification shall describe state's inputs as named entities.

NOTE: This is required for simplifying implementation of transition function (it needs inputs identification) and connections between states.

Logging

FAP_REQ_LOG_01 Logging error on construction phase

Logging shall record errors happend on construction phase

FAP_REQ_LOG_02 Log file specifying

FAP API shall allow requesting the logging into specific log file.

If this is not requested then FAP shall use default log file.

FAP_REQ_LOG_03 Logging in transition functions

Framework shall allow using logging from transition function

FAP_REQ_LOG_04 Logging specification

Framework shall allow to specify logging parameters shall in two ways:

  • In object specification (ref the section called “Object specification”)
  • In dedicated logging specification

Embedded interpreter of transition function

FAP_UC_ET_01 Transition can use the accessible modules

The system defines transition for some state by specifying it within the state. This transition can use the module specifying within the system or super-system.

FAP_UC_ET_02 User can define functions

User defines function and then uses it within transition.

FAP_UC_ET_03 User can define custom data type

User defines custom data type, and operation for this data type

FAP_UC_ET_04 User can specify state of custom data type

User specifies the state of custom data type, ref the section called “FAP_UC_ET_03 User can define custom data type”

FAP_UC_ET_05 User can specify new type

User specifies the new type by specifying the constructor that results in required type.

Connections

UC_CONN_02 Object has its own input that represent input of some internal state

It allows to has the unified interface for states and objects. Morover it solves the problem with access to the object. All access to the object shall be done via objects connection point.

Let's use term "extention" for that. So system connection point is actually an extender. It extends the internal connection point

UC_CONN_03 Connection point can be connected to many connection points

An example is state output that can be connected to many state inputs. The alternative way could be to have many connection points for output

UC_CONN_05 Subsystem connections set can be state of system.

System controls the structure of subsystem (connections set, transition function). The structure is represented as state - it can be connected to other states, thus the change of structure will cause the activation of dependent states

UC_CONN_06 USER removes internal sub-system, SYSTEM disextend connection points of subsystem

  • Initial conditions: There is system with internal subsystem; system has extention that extend connection point of this subsystem.
  • USER request removing internal subsystem
  • SYSTEM automatically disextend extention from subsystem
  • SYSTEM removes systems's subsystem as component.

Views

FAP_UC_VW_01 User specifies view for interpreter.

User runs DES interpreter and specifies how many views should be created and what type they shoud be of.

FAP_UC_VW_02 Client application creates view and adds it to DES interpreter

FAP_UC_VW_06 User specifies base view and see the systems spec

User specify that the view of type "base". System displays it's specification in graphic notation.

System displays the following on the base view: inputs, outputs, transitions, states, components

FAP_UC_VW_03 System shows the DES objects as hierarchy in base view

User specify view and the system set this view as a view of root DES object.

FAP_UC_VW_04 User selects different level of detalisation in view.

System displays the view as a graphical hierarchy. User selects of full or compact form of view. The full form shows inputs, outputs, components (in compact form), and connections. The compact form shows only inputs and outputs of object.

FAP_UC_VW_05 User changes base view and saves - system stores view data and uses it next time.

FAP_UC_VW_07 User can act to view

User acts on view. The system reacts on this and change the view properly.

FAP_UC_VW_09 User taps on connection id

User taps on connecion id - system higlight connection id and all corresponding connection ids

FAP_UC_VW_10 USER selects system with state - SYSTEM displays state, showing state elements

The following elemensts are displayed: name, type, trans, init, inputs. Name and type are displayed in header of state.

FAP_UC_VW_11 USER selects system with sub-systems - SYSTEM displays sub-systems, showing its elements

The following elemensts are displayed: name, parent, inputs, outputs. Name and parent are displayed in the header of system.

FAP_UC_VW_12 USER taps on name in subsystem - SYSTEM displays sub-system view

FAP_UC_VW_13 USER taps on parent in subsystem - SYSTEM displays parent view

FAP_UC_VW_14 SYSTEM allows to open new view in current or new window

FAP_UC_VW_15 SYSTEM allows USER to navigate thru root hierarhy

SYSTEM (for instance Client Application) is initialized by some element within system hierarhy. SYSTEM considers this element as a root. SYSTEM allows USER navigation thru hierarhy below this root.

SYSTEM moves down on USER selects sub-system within current system. SYSTEM moves up on USER selects moving to parent (from menu or toolbar).

FAP_UC_VW_16 SYSTEM allows USER to navigate thru the history

FAP_UC_VW_17 SYSTEM can filter the connections showing only local of them

FAP_UC_VW_18 USER adds new component

New component has no parent and becomes some initial name. USER can create more new components, the names of components are generated uniquely.

FAP_UC_VW_19 USER changes the name of component

FAP_UC_VW_20 USER changes the name of system

FAP_UC_VW_21 USER adds new state

New component has of default type and becomes some initial name. USER can create more new states, the names of them are generated uniquely.

FAP_UC_VW_22 USER removes a state.

All connection from/to this state are deleted correspondingly

FAP_UC_VW_23 USER selects type of state.

FAP_UC_VW_24 USER adds inputs to state.

FAP_UC_VW_25 USER renames input of state.

FAP_UC_VW_26 USER changes transition of state.

FAP_UC_VW_27 USER adds connection

The connection is added between connection points (not to extender)

FAP_UC_VW_28 USER removes connection

FAP_UC_VW_29 USER changes transtition of system.

Studio application

FAP_UC_ST_01 USER creates new project

STUDIO initially manage DES environment, that provides system ticks. STUDIO also provides some predefined projects template. One of them is default template with empty root only.

USER asks STUDIO to create new project. STUDIO displays to user the list of projects templates available. USER selects the proper template. STUDIO creates the new project from the template.

FAP_UC_ST_02 USER saves the system specification.

FAP_UC_ST_03 USER opens system spec from file on local filesystem.

STUDIO initially manages DES environment, that provides system ticks (creates root system).

USER asks STUDIO to open system spec stored as the file on local file system. STUDIO allows USER to select file on filesystem, create system from the spec and add it to the root.
